Engineering Standards — You Will Follow All of These ### 1. TypeScript — Strict Mode, No Exceptions ```json // tsconfig.json { "compilerOptions": { "strict": true, "noUncheckedIndexedAccess": true, "exactOptionalPropertyTypes": true } } ``` - No `any`. Use `unknown` and narrow it. - No `// @ts-ignore`. Fix the actual type error. - Generate Supabase types from schema using `supabase gen types typescript`. Use them everywhere. Never define DB row types manually. --- ### 2. API Routes — Every Single One Follows This Pattern ```typescript // Standard API route template — deviate from this and you've introduced a bug export async function GET(req: Request, { params }: { params: { id: string } }) { // 1. Auth check — always first const session = await getServerSession(authOptions); if (!session?.userId) { return Response.json({ error: { code: 'UNAUTHORIZED', message: 'Authentication required.' } }, { status: 401 }); } // 2. Input validation (Zod for body/params, never trust raw input) const parsed = schema.safeParse(params); if (!parsed.success) { return Response.json({ error: { code: 'INVALID_INPUT', message: parsed.error.flatten() } }, { status: 400 }); } // 3. Ownership check — always verify the resource belongs to this user const resource = await db.getProjectForUser(parsed.data.id, session.userId); if (!resource) { return Response.json({ error: { code: 'NOT_FOUND', message: 'Resource not found.' } }, { status: 404 }); // NOTE: Return 404, not 403 — prevents user enumeration } // 4. Tier check (if applicable) await assertTier(session.userId, 'pro'); // throws TierError if insufficient // 5. Business logic — wrapped in try/catch try { const result = await doTheThing(resource); return Response.json({ data: result }); } catch (err) { return handleApiError(err); // centralised error handler in lib/utils/errors.ts } } ``` Every step in this pattern is mandatory. A route missing any step is an unfinished route. --- ### 3. Database — Migrations First, Always - Every schema change lives in `supabase/migrations/` as a numbered `.sql` file. - Never mutate the DB by hand through the Supabase dashboard during development. - Write migrations to be idempotent where possible (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`). - Always enable RLS on every new table. Write the policies in the same migration file. - Foreign keys always have `ON DELETE CASCADE` unless there is an explicit reason not to. ```sql -- Migration file naming: YYYYMMDDHHMMSS_descriptive_name.sql -- Example: 20241101120000_create_projects_table.sql ``` --- ### 4. Error Handling — Exhaustive, Never Silent Every possible failure mode in PRD Section 14 must be handled. The golden rules: - **No swallowed errors.** A `catch(err) {}` with no action is a silent bug. Always either handle it, log it to Sentry, or re-throw it. - **Distinguish retryable from terminal errors.** GSC 429 = retryable. GSC 403 = terminal. Handle differently. - **User-facing errors must be human-readable.** Never show a raw error message, stack trace, or Supabase error object to the user. - **Background job errors must be persisted.** Any sync failure writes to `sync_jobs.error_message`. If it fails silently, it's invisible. ```typescript // lib/utils/errors.ts — build this early, use it everywhere export class TierError extends Error { constructor(public requiredTier: 'pro' | 'agency') { super(`This feature requires the ${requiredTier} plan.`); this.name = 'TierError'; } } export class OAuthRevokedError extends Error { ... } export class GSCQuotaError extends Error { ... } export class SyncInProgressError extends Error { ... } export function handleApiError(err: unknown): Response { if (err instanceof TierError) { return Response.json({ error: { code: 'UPGRADE_REQUIRED', message: err.message, upgrade_url: '/billing' }}, { status: 403 }); } if (err instanceof OAuthRevokedError) { return Response.json({ error: { code: 'OAUTH_REVOKED', message: 'Please reconnect your Google account.' }}, { status: 401 }); } // ... etc // Unknown error — log to Sentry, return generic 500 Sentry.captureException(err); return Response.json({ error: { code: 'INTERNAL_ERROR', message: 'Something went wrong. We\'ve been notified.' }}, { status: 500 }); } ``` --- ### 5. The OAuth Token Refresh — Get This Right or Everything Breaks This is the highest-risk technical piece. A subtle mistake here causes background syncs to silently die days after launch. **Rule 1:** In the NextAuth `jwt` callback, the refresh token is ONLY present in the first sign-in. Save it immediately to `oauth_tokens`. Every subsequent session callback will NOT have it. **Rule 2:** The `getValidToken()` function in `lib/gsc/token.ts` must: - Check expiry with a 5-minute buffer - Call Google's token refresh endpoint if needed - Save the NEW access token and updated expiry back to the DB - Handle Google's refresh token rotation (when you use a refresh token, Google may issue a new one — save it) - Throw `OAuthRevokedError` if the refresh call returns `invalid_grant` **Rule 3:** Never call GSC API directly anywhere in the codebase. Always go through `getValidToken()` first. No exceptions. **Write a test for this.** Simulate an expired token and verify the refresh happens correctly before any production deployment. --- ### 6. Tier Enforcement — Two Layers, Not One Tier limits are enforced in TWO places: 1. **API layer** — before executing business logic (as shown in the route template above) 2. **UI layer** — to show the right upgrade prompts BEFORE users hit the API limit The UI enforcement is for UX. The API enforcement is for security. Both are required. A UI-only enforcement is a security hole (API can be called directly). An API-only enforcement produces a confusing UX (user doesn't know why something is disabled). ```typescript // UI enforcement example in a React Server Component const session = await getServerSession(authOptions); const isAgency = session?.tier === 'agency'; <Button disabled={!isAgency} onClick={generatePDF}> {isAgency ? Do not simplify it. Do not "start with a basic version." A failure in one must NEVER stop the others. ```typescript // WRONG — one failure kills all subsequent projects for (const project of projects) { await syncProject(project); // if this throws, loop ends } // CORRECT — failures are isolated and logged const results = await Promise.allSettled( batch.map(project => syncProject(project)) ); results.forEach((result, i) => { if (result.status === 'rejected') { logger.error('Sync failed', { project_id: batch[i].id, error: result.reason }); } }); ``` Process projects in batches of 10 with a 2-second pause between batches.
